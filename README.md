- 유저/구매 플로우 (회원가입 → 상품 등록 → 구매 → 주문 조회)

- AZ 장애 시뮬레이션 (노드 drain → 무중단 확인)
- HPA 스케일아웃 (부하 → Pod 자동 확장)
- Istio 카나리 배포 (트래픽 비율 조절, 롤백)
- gRPC 성능 + BFF 병렬 호출 + 동적 검색

## 주요 성과

### WebFlux + Non-blocking 아키텍처

BFF 역할을 하는 게이트웨이에 Spring WebFlux를 적용하여, 내부 서비스 병렬 호출 등에서 Non-blocking 방식을 활용하기도 했습니다.

### Terraform 인프라 자동화

Terraform + EKS로 인프라 전체를 코드화하여, `terraform apply` 한 번으로 자동 생성됩니다.
관리자 계정 초기 데이터 자동 주입도 구현했으나, 아직 검증 전입니다. 나머지는 전부 코드로 관리됩니다.

## 어떤 프로젝트인가요?

실제 서비스 운영 환경과 유사한 구조로 만든 **쇼핑몰 백엔드 시스템**입니다.
사용자 로그인, 상품 조회, 주문 처리 기능을 제공하며,
각 기능을 독립적으로 배포·확장할 수 있도록 **마이크로서비스 아키텍처(MSA)** 로 구성했습니다.

---

## 시스템 구성

```
사용자/관리자
    ↓
[ 게이트웨이 ]  ← 외부에서 유일하게 접근 가능한 입구
    ↓
[ 사용자 서비스 | 주문 서비스 | 상품 서비스 ]  ← 외부 직접 접근 불가
    ↓
[ 각 서비스 전용 데이터베이스 ]
```

---

## 주요 기능 및 적용 기술

### 1. 로그인 보안 — RTR (Refresh Token Rotation)

로그인 시 서버는 단기 열쇠(Access Token)와 장기 열쇠(Refresh Token)를 발급합니다.
RTR은 장기 열쇠로 새 열쇠를 받을 때 장기 열쇠도 함께 교체하는 방식입니다.
한 번 쓴 장기 열쇠는 즉시 폐기되어, 탈취되더라도 재사용이 불가능합니다.

---

### 2. 탈취 감지 및 전체 세션 강제 종료

시스템은 사용자별로 "가장 최근에 발급된 장기 열쇠"를 추적합니다.
이미 교체된 구형 열쇠로 접근을 시도하면 탈취로 판단하여
해당 계정의 **모든 기기 로그인 세션을 즉시 삭제**하고 재로그인을 요구합니다.
탈취 시도 시각과 기기 정보는 서버 로그에 보존됩니다.

---

### 3. 장기 열쇠 쿠키 보안 — XSS·CSRF 방어

장기 열쇠는 자바스크립트에서 접근할 수 없는 **httpOnly 쿠키**에만 저장됩니다.
악성 스크립트로 훔쳐보는 공격(XSS)을 원천 차단하고,
`sameSite=Strict` 설정으로 다른 사이트에서의 위조 요청(CSRF)도 막습니다.

---

### 4. 기기 식별 — 개인정보 보호

로그인 기기를 추적하되 실제 기기 정보는 저장하지 않습니다.
기기 고유값을 **단방향 암호화(HmacSHA256)** 하여 복원 불가능한 형태로만 보관합니다.

---

### 5. 관리자 전용 게이트웨이 분리

사용자용(demoGate)과 관리자용(adminGate) 입구를 완전히 분리했습니다.

관리자 기능: 전체 사용자 조회·검색, 특정 사용자 강제 로그아웃, 계정 삭제

**보안 이중 검증:** 관리자 요청은 게이트웨이(토큰 기반)와 데이터베이스(실제 계정 확인) 두 단계에서 검증되어, 토큰을 조작해도 DB 검증에서 차단됩니다.

---

### 6. 강제 로그아웃 즉시 반영

두 게이트웨이가 **같은 세션 저장소(Redis)를 공유**합니다.
관리자가 특정 사용자의 세션을 삭제하면 사용자 게이트웨이에서도 즉시 반영됩니다.

---

### 7. Google 소셜 로그인

Google 계정으로 로그인하면 계정이 자동 생성·연결됩니다.
일반 로그인과 소셜 로그인 계정이 같은 데이터베이스에서 유연하게 관리됩니다.

---

### 8. 서비스 성격에 맞는 데이터베이스 선택

| 서비스 | 사용 DB | 이유 |
|---|---|---|
| 사용자 서비스 | MongoDB | 소셜/일반 로그인 등 다양한 형태의 데이터에 유연 |
| 주문 서비스 | MySQL | 정확한 트랜잭션 처리 필요 |
| 상품 서비스 | MySQL | 판매자-상품 관계, 재고 동시성 제어 필요 |
| 세션 저장소 | Redis | 빠른 읽기/쓰기, 만료 시간 자동 관리 |

---

### 9. 동시 주문 시 데이터 오류 방지 — 다층 동시성 제어

상황에 따라 서로 다른 방식의 락을 선택해 적용했습니다.

| 상황 | 방식 | 이유 |
|---|---|---|
| 재고 차감 | 비관적 락 | 초과 판매는 절대 허용 불가 → 처리 중 다른 접근 차단 |
| 상품·주문 정보 수정 | 낙관적 락 | 충돌 빈도 낮음 → 충돌 시에만 감지, 평상시 오버헤드 없음 |
| Kafka 중복 주문 | DB 고유값 제약 | 같은 구매 이벤트가 재전달되어도 DB에서 저장 자체 거부 |

Kafka는 네트워크 오류 시 같은 메시지를 재전달할 수 있습니다.
각 구매 요청에 고유 ID를 부여하고 DB 고유값 제약으로 중복 저장을 원천 차단합니다.

---

### 10. 데이터베이스 성능 최적화

**N+1 문제 해결:** 상품 목록 조회 시 상품마다 판매자 정보를 따로 조회하면
쿼리가 상품 수만큼 추가 실행됩니다. JOIN 쿼리로 단일 쿼리 1번에 모두 가져오도록 변경했습니다.
(상품 100건 기준: 쿼리 101번 → 1번)

**인덱스 설계:** 실제 조회 패턴을 분석해 각 서비스에 인덱스를 설계했습니다.
인덱스는 데이터 전체를 뒤지지 않고 원하는 위치로 바로 이동하게 해줍니다.
불필요한 인덱스(실제 쓰이지 않으면서 저장 부담만 늘리는 것)는 제거했습니다.

**복합 인덱스 순서 최적화:** 여러 컬럼을 묶은 인덱스는 앞에 오는 컬럼의 단독 조회에도 사용됩니다.
관리자 검색에서 가장 빈번한 "상태별 조회"를 커버하도록 상태 컬럼을 앞에 배치했습니다.

**구매 경로 최적화:** 구매 시 사용자 활성 상태 확인을 앱 코드가 아닌
DB 조회 조건에 포함시켜, 정상 사용자의 구매를 최소 연산으로 처리합니다.

**동적 검색:** 관리자 유저 검색, 주문·상품 검색은 입력된 조건만 쿼리에 반영됩니다.
QueryDSL(주문), DatabaseClient(상품), ReactiveMongoTemplate(사용자)으로 구현했습니다.

---

### 11. 주문 처리 — Kafka 기반 비동기 처리

구매 요청이 들어오면 즉시 이벤트를 발행하고, 주문 서비스가 비동기로 처리합니다.
사용자는 빠른 응답을 받고, 처리 중 서버가 재시작되어도 이벤트는 유실되지 않습니다.

---

### 12. 상품 조회 성능 — CQRS 패턴

상품 서비스의 **읽기(조회)** 와 **쓰기(등록·구매)** 경로를 분리했습니다.
쓰기는 트랜잭션과 동시성 제어가 필요한 JPA를 사용하고,
조회는 트랜잭션이 불필요하므로 더 빠른 비동기 드라이버(R2DBC)를 사용합니다.
조회 부하가 쓰기 트랜잭션에 영향을 주지 않습니다.

---

### 13. 서비스 간 통신 — gRPC

내부 서비스끼리는 REST 대신 **gRPC**로 통신합니다.
데이터를 이진 형식(Protobuf)으로 압축 전송해 JSON보다 크기가 작고,
통신 규격이 코드로 고정되어 잘못된 데이터를 보낼 수 없습니다.
관리자 유저 검색처럼 결과가 많은 경우 스트리밍으로 순차 전송도 지원합니다.

---

### 14. REST↔gRPC 자동 변환 프록시 — grpc-gateway

상품 서비스는 gRPC로만 통신하지만 외부 클라이언트는 REST를 사용합니다.
Go 언어로 작성된 **product-grpc-gateway**가 REST 요청을 gRPC로 자동 변환합니다.
외부 클라이언트는 REST만 알면 되고, 상품 서비스 내부는 gRPC를 그대로 유지합니다.
이 프록시에도 무중단 종료(Graceful Shutdown)가 적용되어 있습니다.

---

### 15. 주문 조회 — BFF로 네트워크 오버헤드 절감

주문 내역에는 사용자 정보와 주문 목록이 함께 필요합니다.
BFF(게이트웨이)가 없으면 클라이언트가 두 서비스를 따로 호출해야 합니다.

BFF를 통해 클라이언트 요청 1번으로 두 서비스를 **내부에서 병렬 조회**해 합쳐서 반환합니다.
외부 네트워크 요청 2번 → 1번으로 줄어들고, 응답 시간도 단축됩니다.

추가로:
- 토큰 안에 담긴 사용자 ID를 바로 꺼내 쓰므로 신원 확인을 위한 별도 DB 조회가 없습니다.
- 내부 서비스 응답에서 클라이언트에게 불필요한 필드를 제거해 전송 데이터 크기를 줄입니다.
- 병렬 조회로 응답 시간이 단축됩니다. (순차 350ms → 병렬 200ms)

---

### 16. 무중단 종료 — Graceful Shutdown

서버 재배포 시 처리 중인 요청이 강제로 끊기지 않도록 **Graceful Shutdown**을 적용했습니다.
새 요청 수신을 먼저 중단하고, 진행 중인 요청을 모두 완료한 뒤 종료합니다.
실제 테스트에서 서버를 강제 종료하는 동안에도 응답이 끊기지 않음을 확인했습니다.

---

### 17. 트래픽 급증 시 자동 서버 확장 — HPA

접속자가 급증하면 서버(pod)를 자동으로 추가하고, 줄어들면 반납합니다.
클라우드 비용을 낭비하지 않으면서 트래픽 폭증에 자동으로 대응합니다.

---

### 18. 실시간 로그 모니터링 파이프라인

모든 서비스의 로그가 자동 수집되어 한 곳에서 확인 가능합니다.

```
각 서비스 로그 → Fluent Bit → Kafka → Logstash → Elasticsearch → Kibana
```

서버가 잠깐 꺼져도 로그는 버퍼에 보관되었다가 복구 후 자동 전송됩니다. 로그 유실이 없습니다.

---

### 19. 인프라 전체 코드화 — Terraform

서버, 데이터베이스, 네트워크 등 인프라 전체를 코드 파일로 관리합니다.
동일한 환경을 언제든 재현할 수 있고, 변경 이력이 남아 실수 시 되돌리기 가능합니다.
AWS 클라우드에 명령 한 번으로 전체 인프라가 생성됩니다.

---

### 20. 민감 정보 안전 관리 — External Secrets Operator

DB 비밀번호, 인증 키 같은 민감한 정보는 코드에 직접 적지 않고 AWS의 별도 금고(Secrets Manager)에 보관합니다.
별도 관리자(ESO)가 금고에서 꺼내 서버에 전달하고, 정보가 바뀌면 자동으로 갱신합니다.
각 서비스는 금고에 직접 접근하지 않아, 서비스가 해킹되어도 금고 접근은 차단됩니다.
